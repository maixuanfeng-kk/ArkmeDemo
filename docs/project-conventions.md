# 项目工程约定

本约定用于保持代码逻辑清晰、改动可读、文件规模可控。后续每轮开发都应遵守；若确实需要突破限制，必须在迭代日志中说明原因和后续拆分计划。

## 逻辑清晰度

- 单个模块只承担一个清晰职责：数据读写、页面编排、展示组件、业务规则尽量分层。
- 页面组件负责状态编排和用户流程，复杂展示拆成命名明确的小组件。
- 数据规范化、存储迁移、格式化函数放在数据层或纯函数中，不混进 JSX 大段逻辑。
- 命名要表达业务含义，避免 `temp`、`data2`、`handleClick2` 这类弱语义名称。
- 条件分支超过 3 层时优先拆函数或拆组件。
- 同一段 JSX 中重复结构超过 2 次时，优先抽成小组件或配置数组。
- 注释只解释业务意图、迁移原因或不明显的边界，不写“给变量赋值”这类空注释。

## 行数限制

- 单个源码文件目标不超过 300 行，超过 400 行必须拆分或在日志中说明暂缓原因。
- 单个 React 组件目标不超过 120 行，超过 180 行必须拆成子组件或提取 hook。
- 单个普通函数目标不超过 50 行，超过 80 行必须拆成更小的纯函数。
- 单个 JSX return 块目标不超过 90 行，超过后优先提取展示组件。
- 新增功能优先新建聚焦文件，不继续向 `Home.tsx` 这类历史大文件堆逻辑。
- 每轮改动结束前，新增或主要修改的文件需快速自查行数；触碰超限文件时，不做无关大重构，但新增逻辑应尽量外移。

## 迭代要求

- 按最小可用原则交付：先让闭环跑通，再扩展智能化、自动化和复杂边界。
- 每轮只解决当前明确目标，不顺手重构无关模块。
- UI 状态、空态、错误态、加载态要与本轮功能风险匹配。
- 移动端 App 的确认、删除、状态切换等关键操作要优先使用屏幕中心确认弹窗或底部弹层，避免把确认提示藏在滚动内容里。
- 面向用户可见的示例文案必须无歧义：标题写动作，时间写时间，人物写人物，不把不同字段的信息混在一起。
- 涉及日期时间的展示必须来自运行时计算或用户输入，不允许把当前日期硬编码进 UI。
- 手动新建安排里的截止、时间段和提醒时间优先用日期/时间选择控件，不让用户靠自然语言手填时间。
- 当前 Demo 的提醒机制采用 App 内提醒中心，不做系统通知；提醒后的「完成安排 / 以后再说」等状态处理必须保留居中确认。
- 提醒相关 UI 保持低焦虑表达，不使用强烈警告色或“逾期、紧急、失败”这类施压文案。

## 字段边界复盘

- 结构化字段必须守边界：相关人只放人名或人群，地点只放地点，时间只放时间，补充只放说明。
- 归并后的标题必须覆盖所有不同动作；同一安排包含多个动作时，用 ` / ` 这类明显分隔符并列展示，不能只保留第一条来源的标题。
- 标题只展示明确动作，不能把带时间的原话、聊天语气词或上下文句子拼进去；如“下周一下午三点来面试吧”只能落成“面试”。
- 标题清洗不能把明确动作截断成单字动词或半截动作短语；如“打视频”“玩游戏”不能退化成“打”“玩”，“和女朋友打视频”“给妈妈买药”不能退化成“和女朋友打”“给妈妈买”。
- 标题中的包含关系要去重：如“带早餐 / 给小李带早餐”应保留更明确的“给小李带早餐”。
- 多动作安排的详情页必须提供清楚的事项拆分，不能只依赖一行长标题让用户自己理解。
- 未配置提醒时，详情页不要展示“不提前 / 不循环 / 下次提醒”等默认提醒状态，避免单指令安排显得冗余。
- 不能用宽泛正则从自然语言里强行猜字段；如果来源已经有 AI 结构化字段或识别理由里的明确字段，优先相信明确字段。
- 迁移旧数据时，清洗规则只能删除明显污染，不能把地点、店名、介词短语补进相关人。
- 遇到“和妈妈在和平饭店吃饭”这类句子时，“和”是句法连接词，“和平饭店”是地点名；不得把地点拆成“平饭店”或写入相关人。
- 修复智能字段问题时，必须补充同类反例验证：人物 + 地点同句、地点名含“和”、已存在污染数据、正常多人合并。
